# FlagManager 微信小程序需求文档

## 1. 项目概述

### 1.1 项目名称

FlagManager

### 1.2 项目背景

很多人会在学习、健身、考证、阅读、作品创作等场景下立下长期目标，但真正坚持完成的人较少。常见的待办清单更适合短期任务，无法很好地承载长期目标的阶段拆解、持续打卡、反馈总结和奖惩机制。

FlagManager 希望帮助用户管理长期 Flag，通过阶段计划、打卡记录、奖惩机制和可视化统计，让用户更清楚地看到自己的坚持过程。

### 1.3 产品定位

FlagManager 是一款面向个人长期目标管理的微信小程序。它不是普通每日清单工具，而是围绕“长期目标的建立、推进、记录和复盘”展开。

### 1.4 核心目标

- 帮助用户记录长期 Flag。
- 支持将长期 Flag 拆分为多个阶段计划。
- 通过打卡记录阶段执行情况。
- 通过奖惩机制增强用户的执行动力。
- 通过类似 GitHub Contributions 的视图展示坚持情况。
- 为后续 AI 检测与阶段总结报告预留扩展空间。

## 2. 目标用户

### 2.1 主要用户

- 大学生
- 自学者
- 考研、考公、考证人群
- 健身、阅读、写作等长期习惯培养者
- 有长期项目或作品计划的个人开发者、创作者

### 2.2 典型场景

- 用户希望半年内完成某门课程学习。
- 用户希望三个月内坚持健身。
- 用户希望一年内读完若干本书。
- 用户希望完成一个个人项目。
- 用户希望记录长期计划中的阶段性成果。

## 3. 产品范围

### 3.1 当前阶段范围

当前阶段优先实现 MVP，不实现复杂 AI 能力和付费体系。

MVP 重点包括：

- Flag 创建与管理
- 阶段计划管理
- 打卡记录
- 阶段奖惩设置
- 打卡统计视图
- 基础个人中心

### 3.2 暂不实现内容

- AI 检测打卡内容真实性
- AI 生成阶段性总结报告
- VIP 付费功能
- 社交广场
- 好友监督
- 排行榜

这些能力作为后续版本规划。

## 4. 核心功能需求

### 4.1 Flag 管理

用户可以创建一个长期 Flag。

每个 Flag 包含：

- Flag 名称
- Flag 描述
- 开始日期
- 目标完成日期
- 所属分类
- 当前状态
- 可选封面或图标

Flag 状态包括：

- 进行中
- 已完成
- 已放弃
- 已暂停

用户可以对 Flag 进行编辑、暂停、恢复、完成或放弃操作。

### 4.2 阶段计划管理

一个 Flag 可以包含多个阶段计划。

每个阶段计划包含：

- 阶段名称
- 阶段目标
- 开始日期
- 结束日期
- 打卡频率
- 奖励内容
- 惩罚内容
- 阶段状态

阶段状态包括：

- 未开始
- 进行中
- 已完成
- 未完成

阶段计划用于把长期目标拆解成更容易执行的小目标。

### 4.3 打卡功能

用户可以在阶段计划下进行打卡。

每条打卡记录包含：

- 所属 Flag
- 所属阶段
- 打卡日期
- 打卡文字内容
- 可选图片
- 当前心情或状态
- 创建时间

打卡内容用于记录用户当天或当次对阶段计划的执行情况。

当前阶段只做普通记录，不做 AI 真实性检测。

### 4.4 奖惩机制

用户可以为每个阶段计划设置奖励和惩罚。

示例：

- 完成阶段后奖励自己看一场电影。
- 连续未打卡三天后减少娱乐时间。
- 完成一周学习后奖励一次聚餐。

MVP 阶段只记录奖惩内容，不强制执行。

后续可以扩展为：

- 阶段完成后提醒用户兑现奖励。
- 阶段失败后提醒用户执行惩罚。
- 引入监督人或好友见证。

### 4.5 打卡统计视图

系统需要提供类似 GitHub Contributions 的打卡热力图。

视图特点：

- 按日期展示打卡情况。
- 不同颜色深浅表示打卡数量或完成程度。
- 支持查看某一天的打卡记录。
- 支持按 Flag 或全部 Flag 筛选。

统计数据包括：

- 累计打卡天数
- 当前连续打卡天数
- 最长连续打卡天数
- 本周打卡次数
- 本月打卡次数
- 阶段完成率

该功能是产品的重要视觉反馈点。

### 4.6 首页

首页用于展示用户当前最重要的长期目标。

首页内容包括：

- 正在进行的 Flag 列表
- 今日可打卡阶段
- 最近打卡记录
- 当前连续打卡天数
- 快速打卡入口

首页应避免变成普通待办清单，重点展示长期目标的推进状态。

### 4.7 Flag 详情页

Flag 详情页展示单个 Flag 的完整信息。

页面内容包括：

- Flag 基本信息
- 当前进度
- 阶段计划列表
- 打卡热力图
- 最近打卡记录
- 奖惩信息

用户可以从该页面进入阶段详情、添加打卡或编辑 Flag。

### 4.8 阶段详情页

阶段详情页用于查看某个阶段计划的执行情况。

页面内容包括：

- 阶段目标
- 阶段时间范围
- 当前阶段状态
- 打卡记录列表
- 奖励与惩罚内容
- 阶段统计数据

### 4.9 个人中心

个人中心提供基础账户和数据入口。

页面内容包括：

- 用户昵称与头像
- 总 Flag 数
- 已完成 Flag 数
- 累计打卡天数
- 设置入口
- 关于项目

## 5. AI 功能规划

AI 功能当前阶段不实现，但需要在产品设计中预留位置。

### 5.1 AI 打卡检测

目标：

对用户提交的打卡内容进行简单判断，识别内容是否与阶段目标相关。

可能能力：

- 判断打卡内容是否过于敷衍。
- 判断内容是否与当前阶段目标相关。
- 给出简单改进建议。

示例：

用户阶段目标是“每天学习英语 1 小时”，打卡内容为“今天学习了 30 个单词，并完成一篇阅读”。AI 可以判定为有效打卡。

### 5.2 AI 阶段总结报告

目标：

根据阶段内的打卡记录生成总结报告。

报告内容可能包括：

- 阶段完成情况
- 高频关键词
- 用户坚持情况
- 做得好的地方
- 后续改进建议

该功能可作为后续 VIP 功能。

## 6. VIP 功能规划

当前阶段不实现 VIP，但可以作为商业化方向。

候选 VIP 功能：

- AI 阶段总结报告
- 更长时间范围的数据统计
- 更多 Flag 数量
- 更多图片存储空间
- 高级热力图分析
- 数据导出
- 自定义主题

MVP 阶段应避免过早引入复杂付费逻辑。

## 9. 非功能需求

### 9.1 易用性

- 创建 Flag 和打卡流程应尽量简单。
- 首页应突出当前最重要的目标。
- 统计视图应直观清晰。

### 9.2 数据安全

- 用户只能访问自己的 Flag 和打卡数据。
- 后续如接入 AI，需要注意用户打卡内容的隐私保护。

### 9.3 性能

- 首页和统计页需要快速加载。
- 热力图数据应按时间范围查询，避免一次加载过多历史数据。

## 10. MVP 优先级

### 10.1 P0 必须实现

- 创建、编辑、删除 Flag
- 创建、编辑、删除阶段计划
- 添加打卡记录
- 查看 Flag 详情
- 查看阶段详情
- 基础打卡统计

### 10.2 P1 优先实现

- GitHub 风格打卡热力图
- 连续打卡天数统计
- 奖惩内容记录
- 分类筛选

### 10.3 P2 后续实现

- AI 打卡检测
- AI 阶段总结
- VIP 功能
- 数据导出
- 好友监督

## 11. 开发建议

### 11.1 技术方向

建议优先采用微信小程序原生开发或 uni-app 开发。

如果希望快速上线 MVP，可以使用：

- 微信小程序原生框架
- 微信云开发
- 云数据库
- 云函数

### 11.2 开发顺序

建议开发顺序：

1. 搭建小程序基础结构。
2. 实现用户登录与基础用户信息。
3. 实现 Flag 管理。
4. 实现阶段计划管理。
5. 实现打卡记录。
6. 实现统计视图。
7. 优化页面体验。
8. 再考虑 AI 和 VIP 扩展。

## 12. 成功标准

MVP 阶段完成后，用户应能够：

- 创建一个长期 Flag。
- 将 Flag 拆分为阶段计划。
- 在阶段计划中持续打卡。
- 查看自己的打卡记录和统计视图。
- 通过奖惩内容提醒自己坚持目标。

如果用户能够清楚看到“我为了这个长期目标坚持了多久、做了什么、现在进展如何”，则 MVP 达到目标。
